## 0. 한 줄 정의

> **포트폴리오 완성작을 들고 온 디자이너에게, 6명의 회사 스타일 페르소나가 면접·리뷰에서 받을 만한 질문과 피드백을 주는 웹 앱.**

**1차 사용 시나리오**: 포트폴리오 완성작 피드백 (면접·리뷰 전 자기 표현 리허설)
**부수 시나리오**: 회사 진행 중 작업 피드백 (도구로는 가능, 디폴트 톤은 포트폴리오)
**타겟 사용자**: 자기 어필이 어려운 한국 프로덕트 디자이너 (특히 주니어·미드)
**핵심 가치**: 평가가 아닌 동료의 피드백 + 자기 표현의 리허설.

### 2.2 디자인 원칙 충돌 풀이 (페르소나 표기 규칙)

원래 PRD 원칙: *"회사 로고는 쓰지 않는다. 페르소나는 관점이지 브랜드가 아니다."*
실명 회사 사용과 충돌 → **풀이 2 채택**: `"○○ 스타일"` 접미사 + 디스클레이머.

**디스클레이머 문구 (UI에 명시 필요)**:
> 각 회사의 공개된 디자인 철학과 블로그/컨퍼런스 발언을 기반으로 재구성한 가상의 페르소나입니다. 실제 해당 회사 또는 직원의 의견을 대변하지 않습니다.

**금지**: 회사 로고/색/폰트 차용. 텍스트 레이블만.

## 4. ✅ 확정: UX·콘텐츠 결정

### 4.1 충돌 트리거 정책

**채택**: **A — 매트릭스 룩업 자동 트리거** (V2에서 D 하이브리드로 진화)

- 사용자가 페르소나 선택 → 그중 강한 충돌 쌍이 있으면 자동으로 충돌 카드 등장
- 추가 API 호출 0
- 매트릭스 정의 **노출** (대화적 톤): "토스 스타일 PO와 우아한 스타일 CBO는 자주 충돌하는 관점이에요. 이번 디자인에서도 그럴까요?"

**기각된 옵션**:
- B (AI 자동 감지): API 호출 +1, MVP에 과함 — V2 후보
- C (유저 클릭): USP 묻힘
- D (A+B 하이브리드): V2 후보

### 4.2 맥락 대화 4개 질문 (포트폴리오 톤)

스샷 업로드 후, 페르소나 크리틱 받기 전에 사용자가 답하는 질문. 객관식 2 + 자유 2.

| | 질문 | 형식 |
|---|---|---|
| ① | **이 작업의 종류는?** | 객관식: `실무 출시작` / `실무 컨셉` / `사이드 프로젝트` / `학생 작품` / `리디자인` / `기타` |
| ② | **이 작업이 해결하려던 핵심 문제는?** (한 문장) | 자유 (50자 이내 권장) |
| ③ | **본인의 역할은?** | 객관식: `PD 단독` / `PD + PM` / `디자인팀 일부` / `기타` |
| ④ | **이 작업에서 가장 자랑하고 싶은 결정은?** (없으면 비워도 OK) | 자유 (옵션) |

→ 면접·포트폴리오 리뷰에서 자주 받는 질문 그 자체. 이걸 미리 답해보는 것 자체가 리허설.
→ ②와 ④는 디자이너의 자기 의도 언어화 첫 연습.

### 4.3 크리틱 출력 포맷 (페르소나 카드 1개)

**구조**: 한 줄 진단 + 질문 3개 + 제안 1-2개

**가드레일 (system prompt에 박제)**:
| 항목 | 길이 제한 |
|---|---|
| 한 줄 진단 | 40자 이내 |
| 질문 (각) | 50자 이내 |
| 제안 (각) | 80자 이내 |
| **카드 전체** | **약 200자 이내** (모바일 한 스크린에 2장) |

**질문 톤 원칙**: 페르소나 본인 머릿속 언어(전환율, MAU, 법적 필수 같은) ❌ → 디자이너가 답할 수 있고 디자인 결정으로 연결되는 디자이너 언어 ✅

**샘플 (토스 PO 페르소나, 가상 디자인: 카드 발급 약관 7개 화면)**:
> **🩺 한 줄 진단**: 7개 약관이 다 같은 시각적 무게라, 사용자가 어디서 멈춰야 할지 모를 거야.
> **❓ 질문**
> 1. 이 화면에서 사용자 시선이 가장 먼저 가는 곳이 어디야?
> 2. 필수와 선택을 시각적으로 어떻게 구분했어?
> 3. "전체 동의"와 "발급 신청" 버튼의 위계 차이가 보여?
> **💡 제안**
> - 필수 3개를 한 그룹(배경/구분선)으로, 선택 4개는 보조 영역으로 분리
> - "전체 동의"를 발급 버튼 바로 위에 같은 폭으로

### 4.4 디자인 원칙 (7개 확정)

1. **크리틱은 판결이 아니라 대화다.**
2. **답이 아니라 질문에 무게중심을 둔다.** (제안은 허용하되 강한 자리는 질문)
3. **여백을 두려워하지 않는다. 침묵도 피드백이다.**
4. **회사 이름은 빌리되 로고·색·폰트는 차용하지 않는다.** (페르소나는 관점이지 브랜드가 아니다)
5. **디자이너에게는 디자이너 언어로 말한다.** (PO·센터장 머릿속 언어가 아니라 디자인 결정의 언어로 변환)
6. **평가가 아니라 동료의 피드백이다. 완성작이 다음 작업으로 이어지도록 본다.**
7. **자기 표현의 리허설이다. 디자이너가 자기 작업을 더 잘 이야기하게 돕는다.**

→ 7원칙 모두 system prompt에 들어가서 페르소나 응답에 직접 영향. 자구 한 글자도 무게 있음.

### 4.5 페르소나 톤 가이드 (system prompt 공통 부분)

> 디자이너가 **완성된 작업을 포트폴리오로 들고 와서 보여줄 때**, 시니어 동료/포트폴리오 리뷰어 입장에서 봅니다. 회의실 동료가 아니라 첫 만남의 리뷰어. **디자이너 언어로 말하고**, 면접·리뷰에서 받을 만한 질문을 던집니다.
>
> - 평가하지 않는다. 동료처럼 본다.
> - 자기 머릿속 비즈니스 언어(전환율, MAU, 법적 필수 등)로 묻지 않는다. 디자이너가 답할 수 있는 디자인 결정의 언어(시각적 위계, 그룹핑, 시선 흐름, 컴포넌트 일관성 등)로 변환해서 묻는다.
> - 질문 → 제안 순서. 답은 강요하지 않는다.
> - 자기 영역(§2.1의 질문 영역)을 벗어나는 주제는 다른 페르소나에게 양보한다.

### 4.6 유저 플로우 5단계 (1차 가안 — 구현 후 디테일 다듬기)

**원칙**: 구조에 영향 큰 결정만 지금 박제. 시각적 디테일(레이아웃, 애니메이션, 카드 모양)은 MVP 짓고 나서 보면서 다듬는다.

#### STEP 1 — 랜딩

- **헤드라인**: "완성작 들고 와요. 6명이 봐줍니다." (부록 카피 후보 1번)
- **1차 액션**: 스크린샷 드래그앤드롭 영역 (단일 칸)
- **이미지 처리**: **다중 업로드 허용** (포트폴리오는 보통 여러 화면) — 구현 후 UX 검토
- **부가**: 디스클레이머(§2.2) 한 줄 + 페르소나 6인 이름 노출 (호기심 트리거)

#### STEP 2 — 맥락 대화

- **위치**: 한 페이지에 4개 질문(§4.2) 모두 노출
- **레이아웃**: 좌측 스샷 미리보기 / 우측 질문 폼 (데스크톱 기준, 모바일은 세로 스택) — 구현 후 검토
- **진행 버튼**: "6명에게 보여주기"

#### STEP 3 — 페르소나 선택

- **디폴트**: **6명 전체 선택** (디폴트 첫 경험 = 풀세트) — 구현 후 3명 디폴트와 비교 검토
- **선택 해제 가능**: 체크박스로 빼기 가능
- **선택 화면 하단**: 충돌 미리보기 — "선택한 페르소나 중 강한 충돌 N쌍이 있어요" (§4.1 노출 톤)

#### STEP 4 — 크리틱 생성 (메인 결과 화면)

- **방식**: 6명(또는 선택 인원) **병렬 스트리밍**
- **등장**: 페르소나별 카드가 응답 도착 순서대로 등장 (스트리밍 텍스트)
- **레이아웃**: 모바일 세로 스크롤 / 데스크톱 2열 그리드 — 구현 후 검토
- **카드 콘텐츠**: §4.3 출력 포맷 (한 줄 진단 + 질문 3개 + 제안 1-2개, 약 200자)

#### STEP 5 — 충돌 뷰 + 자기 표현 리허설 ⭐

- **위치**: 같은 페이지 아래 자동 등장 (페이지 이동 없음)
- **트리거**: §4.1 매트릭스 룩업으로 강한 충돌 쌍 자동 카드화
- **카드 구조**:
  - 충돌 테마 (§3.2): "전환율 vs 기분 좋음, 어느 쪽?"
  - 두 페르소나 각자의 한 줄 입장
  - **"당신은 어느 쪽?" 입력칸** ✅ (확정 — USP 핵심)
- **입력칸 의미**: 면접·포트폴리오 리뷰에서 답해야 할 질문을 미리 답해보는 리허설. 입력은 클라이언트 저장만(MVP 단계, DB 없음 가정).
- **선택성**: 답하지 않아도 됨 (강제 아님)

#### 구현 후 검토 항목 (지금 결정 안 함)

| 항목 | 1차 가안 | 보면서 다듬을 것 |
